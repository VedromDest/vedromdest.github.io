Requirements

  • Het systeem kent weerstations

    • Een weerstation kan metingen doen

  • Het systeem stelt een http api ter beschikking

Weerstation

Een weerstation kent

  • Locatie (=stad object)

  • Lijst van gedane metingen

Er zijn vier soorten stations

  • Luchtdruk

  • Neerslag

  • Temperatuur

  • Wind

Een weerstation kan gevraagd worden

  • kopie van lijst van gedane metingen teruggeven

  • een meting te doen

    • implementeer het genereren van een passende meting met willekeurige waarde binnen realistische grenzen.

Logging

Elke meting wordt gelogd door een component die zich abonneert op nieuwe meting-events van alle weerstations.

Er is een basis implementatie die het meting-event als string op de console logt. ‚úÖ

Deze kan decorated worden met XML en/of Json functionaliteit. Bij het starten van de applicatie wordt bepaald welke logging(decorator(s)) actief zijn. Dit heeft geen verdere impact op het systeem.

De logging component wordt aangemaakt met behulp van een factory.

De XML en Json decorators appenden respectievelijk een XML en Json representatie aan hun eigen log files log.xml en log.json.

Voorbeeld XML log entry
<Meting>
	<Moment>7/05/2025 00:54:20</Moment>
	<Waarde>82</Waarde>
	<Eenheid>kmh</Eenheid>
</Meting>
Voorbeeld Json log entry
{
  "Moment": "2025-05-07T00:54:21.079788+02:00",
  "Eenheid": "kmh",
  "Waarde": 31,
  "Locatie": {
    "Naam": "Gent",
    "Beschrijving": "Grootste stad van Oost-Vlaanderen",
    "GekendVoor": "Studenten"
  }
}
Je mag gewoon de log entries gewoon appenden aan de files. Je hoeft deze niet globaal te omsluiten in een [] (json) of <root></root> (xml).

Weerbericht

Een weerbericht kent

  • Moment van creatie

  • Tekstuele inhoud

Er bestaat een systeemcomponent die zich abonneert op meting-events. Deze worden gecombineerd met een weermodel om tot een weerbericht te komen. Dit is een zware operatie - we simuleren dat met Thread.Sleep(5000)

De component voor logging en de component voor weerberichten creatie staan totaal los van elkaar!

Inhoud Weerbericht: Op basis van {n} metingen en mijn diepzinnig computermodel kan ik zeggen dat er kans is op {goed|slecht} weer.

Gezien het maken een weerbericht zo veel tijd kost, plaatsen we een proxy voor de klasse die verantwoordelijk is voor het maken van een weerbericht. De proxy zal lazy loading doen en het gemaakt weerbericht gedurende 1 minuut cachen. De aanvrager van een weerbericht weet niet expliciet dat hij tegen een proxy aan het spreken.

Facade

Domeincontroller + DTOs schermen de interne keuken volledig af. De http api route mapping code maakt enkel gebruik van de domeincontroller en diens DTOs.

Http Api

  • GET / ‚úÖ

    • geef een server-is-running status boodschap

  • GET /steden ‚úÖ

    • geef alle steden

  • GET /weerstations

    • geef alle stations (zonder de metingen)

  • GET /metingen

    • geef alle metingen

  • GET /weerbericht

    • geef weerbericht

  • POST /commands/meting-command

    • alle stations worden gevraagd een meting te doen

Bootstrapping / Flow

  • Gebruik .net dependency injection om alle externe afhankelijkheden als singleton binnen te brengen in onder andere domeincontroller.

  • Er worden 12 willekeurige weerstations aangemaakt door een factory.

  • De app draait volledig in memory. Er is geen database. Bij opstart zijn er dus geen metingen beschikbaar.
